# Contributing

Thanks for your interest in contributing. This document describes how to report issues, propose changes, and get a local development environment running.

## Reporting issues

- Search [existing issues](../../issues) before opening a new one.
- For bugs, include: Obsidian version, Oxygen version, operating system, reproduction steps, and what you expected vs. what happened. Screenshots help.
- For feature requests, describe the use case and the problem the feature would solve, not just the proposed solution.

## Development setup

Requirements:

- [Node.js](https://nodejs.org/) (LTS recommended)
- [pnpm](https://pnpm.io/). This repo pins a specific version via `packageManager` in `package.json`.

```bash
pnpm install
pnpm exec grunt          # watch mode: compiles on .scss/.css changes
pnpm exec grunt build    # production build (theme.css + Oxygen.css)
pnpm lint                # check style
```

Build artifacts (`theme.css`, `Oxygen.css`) are produced at the repo root from `src/scss/`. To test against a real vault, point the build output at `<vault>/.obsidian/themes/Oxygen/` (symlink `theme.css`, or use the watch task with an `OBSIDIAN_PATH` set in `.env`).

## Source layout

- `src/scss/` — SCSS source (organized by `app/`, `content/`, `features/`, `plugins/`, `color-schemes/`, `variables/`)
- `src/scss/index.scss` — the SCSS entry point, imports everything
- `src/css/` — intermediate build outputs
- `theme.css` / `Oxygen.css` — final concatenated theme files at the repo root

## Pull requests

- Open an issue first for anything non-trivial so we can agree on direction before you spend time on it.
- Keep PRs focused. One concern per PR is easier to review.
- Edit **`src/scss/`** files, not the generated `theme.css` / `Oxygen.css` — those will be regenerated by the build.
- Run `pnpm exec grunt build` before submitting so the generated files are up to date.
- Follow the existing code style. Use **"properties"** (not "frontmatter") when referring to YAML metadata. **"Markdown"** is always capitalized.
- Update the README if you change user-facing behavior or default style settings.

## Releases

Maintainers cut releases by bumping the version in `manifest.json`, `package.json`, and `versions.json`, committing, then tagging and pushing the tag. The release workflow builds the theme and attaches `theme.css`, `Oxygen.css`, and `manifest.json` with build provenance attestation.
